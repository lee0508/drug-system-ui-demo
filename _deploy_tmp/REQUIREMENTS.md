## 1. 문서 개요

### 1-1. 목적

본 문서는 마약류중독자관리시스템의 **기능적·비기능적 요구사항**을 체계적으로 정의하여, 이해관계자 간 공통된 이해를 확보하고 설계·개발의 기준점으로 활용하는 것을 목적으로 한다.

### 1-2. 분석 방법

| 분석 자료 | 내용 |
| --- | --- |
| `1.html` (시스템 개념도) | 6대 기능 영역·인프라·표준프레임워크 구성도 |
| `lib/types.ts` | 도메인 모델 (Intake, Case, ISPPlan, IntegrationConnector 등 9개 타입) |
| `lib/mockData.ts` | 실 업무 시나리오 반영 샘플 데이터 (접수·사례·연계 커넥터·로그) |

## 2. 시스템 범위 및 목적

### 2-1. 시스템 정의

마약류중독자관리시스템은 마약류 중독자의 **재활·치료 지원 전 주기(접수→사례관리→모니터링→종결)**를 전산화하는 공공 행정 시스템으로, 전국 마약류 중독자 재활센터의 업무를 표준화하고 국가 통계 보고를 자동화한다.

### 2-2. 시스템 경계

```text
                     ┌─────────────────────────────────────────┐
                     │        마약류중독자관리시스템              │
                     │                                          │
1342 콜센터  ────────▶│  접수/초기개입  →  사례관리  →  통계      │
온라인 신청  ────────▶│                                          │
직접 방문    ────────▶│  대상자 관리        공통(Ops)            │
기관 의뢰    ────────▶│                                          │
                     └──────────┬─────────────────┬────────────┘
                                │                 │
                  ┌─────────────▼───┐   ┌─────────▼─────────┐
                  │ EA 시스템        │   │  행정 기관 시스템    │
                  │ - 마약류통합관리  │   │  - 법무부           │
                  │ - 전문인력 인증  │   │  - 검찰청           │
                  └─────────────────┘   │  - 경찰청           │
                                        │  - MHIS(정신건강)   │
                                        └───────────────────--┘
```

### 2-3. 시스템 외부 조건 (현행 프로토타입 확인)

현재 Next.js 프로토타입은 **완전한 인-메모리 SPA**로 동작한다.

- 데이터 저장: `useState` (새로고침 시 소실)
- 인증: 없음 (Shell 컴포넌트에 "RBAC에 따라 제한될 수 있습니다" 주석만 존재)
- 외부 API: 없음 (커넥터·로그 모두 mockData)
- 대상자(Subject) 마스터: 없음 (사례에 가명 `CASE-XXXXXX`만 존재)

## 3. 이해관계자 분석

| 이해관계자 | 역할 | 주요 관심사 |
| --- | --- | --- |
| **센터 담당자** (사례관리사) | 대상자 접수·상담·ISP 수립·모니터링 | 쉬운 UI, 업무 효율, 중복 입력 최소화 |
| **센터 관리자** | 센터 내 인력·통계·마감 관리 | 현황 파악, KPI 모니터링, 권한 통제 |
| **권역 센터** | 여러 센터 통합 관리 | 권역 내 현황 집계, 기관 간 의뢰·전원 |
| **1342 콜센터 상담원** | 전화 접수·초기 스크리닝 | 신속한 접수 처리, 긴급도 판단 |
| **정부 부처 담당자** | 국가통계 보고 수신·정책 분석 | 표준화된 정기 보고, 다차원 분석 |
| **시스템 관리자** | 권한·인프라·연계 관리 | 가용성, 보안, 연계 모니터링 |
| **보안/감사 담당자** | 개인정보·감사 준수 | 로그 추적성, 암호화, 법령 준수 |
| **법무부·검찰·경찰청** | 이수명령·기소유예 정보 제공 | 정확한 이수 확인, 데이터 신뢰성 |

## 4. 현행 프로토타입 분석 및 갭 분석

### 4-1. 구현 완료 기능

| 기능 | 구현 내용 | 주요 타입/로직 |
| --- | --- | --- |
| 접수 대기열 | 3개 mock 접수, 소스 필터, 선택 | `Intake` type |
| 초기개입 5단계 스테퍼 | QUEUE→SCREEN→CONSENT→ASSIGN→CONVERT UI | `steps` 배열 |

| 사례 목록 | 상태/지역/긴급도 표시 | `Case[]` |
| 사례 상세 5탭 | 요약·타임라인·업무·모니터링·종결 탭 | `CaseEvent[]`, `CaseTask[]` |
| ISP 수립 | 문제·목표(단/중/장기)·개입·검토주기·위기계획 | `ISPPlan` |
| 타임라인 이벤트 등록 | 이벤트 타입 7종 추가 가능 | `addCaseEvent()` |
| 연계관리 현황 | 6개 외부 커넥터 상태(OK/DEGRADED/DOWN) | `IntegrationConnector` |

| 기초통계 차트 | MiniChart (SVG 선 그래프) | `MiniChart` 컴포넌트 |
